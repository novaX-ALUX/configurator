# novaX Configurator M1 Implementation Plan (rev2, revised per Codex review)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver M1 of the novaX Configurator: browser-side MAVLink core stack + connection top bar (including STATUSTEXT/heartbeat debug panel) + parameter table + firmware update page (normal serial update + F4 DFU recovery), plus manifest generation on the flight_controller side. The full MAVLink console moves to M2 (aligned with the spec milestones).

**Context:** Based on the approved spec (`GC/docs/superpowers/specs/2026-07-02-novax-configurator-design.md`) and the Codex plan review (2026-07-02, session `019f20fc-...674f`). Effort calibration: roughly 8–12 weeks for one person full-time; if compression is needed, see "Explicitly out of scope for M1" for the cut order.

**Architecture:** React 18 + Vite + TS SPA; core (transport / mavlink / firmware) is layered separately from features; the message-definition provider is locked in by the Phase 0 spike (mavgen TS preferred, mavlink-mappings version-pinned as fallback); the firmware manifest is generated by the flight_controller release pipeline. **Execution order (Codex recommendation): scaffolding → spikes and lock in decisions → manifest data contract → MAVLink core → feature pages.**

**Tech Stack:** React 18, Vite, TypeScript, Zustand, Tailwind CSS, i18next (full en, zh/ko/ja only need keys to exist), Vitest; Node SITL bridge; Python (gen_manifest.py).

## Global Constraints (excerpted from spec, apply to every task by default)

- Browser runtime must not use Node-only APIs (`Uint8Array/DataView`, not `Buffer`); do not introduce the `node-mavlink` runtime
- Parameter writes must be confirmed by reading back `PARAM_VALUE`; dangerous commands must not be blindly retransmitted
- Hard gate before erase: bootloader board_id == `.apj` board_id and sha256 passes; `AUTOPILOT_VERSION` is display-only, **not used for any gating or list filtering** (AF-F4_T10 does not return this message)
- Software-triggered DFU entry is F4-series novaX boards only; DFU full-chip erase must pass the chip guard (capacity/family) first
- Destructive flows (erase/program) are encapsulated inside the state machine; the public API cannot bypass the guard
- Dark theme, desktop 1280px; pages are implemented as wireframes first, visual spec pending the parallel UI/UX design mockups
- npm, Vitest, conventional commits

**Explicitly out of scope for M1** (Codex cut list): the full MAVLink console, PWA install polish, complete zh/ko/ja translation, per-component fine-grained packet-loss stats, the virtualized-scroll message browser, parameter metadata (pdef).

## Phase 2 — MAVLink core (TDD; fixture `.bin + expected.json` checked into the repo, CI does not depend on pymavlink)

### Task 2.1: Transport abstraction

**Files:** Create: `src/core/transport/{types.ts,serial.ts,websocket.ts,mock.ts}`;Test: `src/core/transport/__tests__/`

**Produces:**
```ts
interface Transport {
  open(opts?: { signal?: AbortSignal }): Promise<void>
  close(): Promise<void>                                  // idempotent
  readonly readable: ReadableStream<Uint8Array>            // one-shot stream available after open, lifetime tied to the connection
  write(data: Uint8Array): Promise<void>                   // must reject if called after close
  onDisconnect(cb: (reason: string) => void): () => void   // returns an unsubscribe function
}
class SerialTransport implements Transport { constructor(port: SerialPort, baud: number) }
class WebSocketTransport implements Transport { constructor(url: string) }
class MockTransport implements Transport { feed(bytes: Uint8Array): void; sent: Uint8Array[] }
```
- [ ] A shared contract test suite reused across all three implementations, covering: chunked arrival, stream-end semantics after close, disconnect mid-read, write rejection, idempotent close, unsubscribe
