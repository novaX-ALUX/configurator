# novaX Configurator

A modern, browser-based setup tool for **ArduPilot** flight controllers. Plug a board in over USB, pick its serial port, and configure, calibrate, test, and flash it — no install, no drivers on the normal-update path. Built for the [novaX-ALUX](https://github.com/novaX-ALUX) AF-series flight controllers, but it works with any ArduPilot board; novaX-specific enhancements (online firmware list, one-click software DFU) unlock only for novaX board IDs.

> **Status:** M1 and M2 are implemented and merged (697 tests, verified in unit tests and against ArduPilot SITL). **Not yet verified on real hardware and not yet deployed.** See [Before shipping](#before-shipping).

The tool talks **MAVLink** over the **Web Serial API** and flashes firmware over **Web Serial** (PX4 serial bootloader) or **WebUSB** (STM32 DFU). It is ArduPilot-only — it does not speak MSP, so Betaflight configuration is out of scope (use BF Configurator for that).

## Features

| Page | What it does |
|------|--------------|
| **Connect** | Web Serial port picker, link status, board/firmware identity (display-only) |
| **Dashboard** | Live 2D attitude, armed/mode, voltage/current, GPS, RC channels, motor outputs (read-only telemetry) |
| **Setup** | Frame class/type, ESC protocol, battery monitor, failsafes — staged, reviewed, then written with read-back confirmation |
| **Calibration** | Accelerometer 6-face + compass, with a **before/after review gate** — no compass parameter is written until you see the diff and confirm |
| **Motor Test** | Per-motor test + sequence, full 0–100% range with a settable 1–30 s spin, behind a prop-removal gate and six independent kill-switches that each send a real stop to the flight controller |
| **Parameters** | Full parameter table: search, group, stage edits, diff drawer, batch write with per-row verification |
| **Firmware** | Online (novaX boards) or local-file flashing; normal driverless serial update + DFU recovery; board-ID and SHA-256 hard gates before any erase |
| **Console** | STATUSTEXT stream + link stats (debug panel) |
| **Setup Guide** | Read-only onboarding checklist (never changes parameters) |

**Safety is the point.** Parameter writes are always read back and confirmed; calibration never writes silently (the compass review gate exists because a competitor tool once corrupted users' compass configs); motor testing is gated and every exit path stops the motors. Firmware flashing refuses to erase unless the bootloader's board ID matches the firmware's and the SHA-256 verifies.

## Tech stack

- **React 18 + Vite + TypeScript**, Zustand state, Tailwind CSS
- **i18next** — English (source), Chinese, Korean, Japanese
- **Hand-written MAVLink2 stack** (frame parser, router, command/ACK, parameter protocol, telemetry) over browser-native `Uint8Array`/`DataView` — no Node polyfills
- Message definitions from [`mavlink-mappings`](https://www.npmjs.com/package/mavlink-mappings) (LGPL), isolated behind a single adapter (see [Before shipping](#before-shipping))
- **Vitest** for tests; ArduPilot SITL for integration

No runtime dependency beyond the six above. Desktop Chrome/Edge only (Web Serial / WebUSB are desktop-only).

## Architecture

```
src/core/transport/   Web Serial / WebSocket(SITL) / Mock transports + shared contract
src/core/mavlink/     frame · router · command · params · telemetry · accelCal · magCal · defs(adapter)
src/core/firmware/    PX4 serial bootloader · WebUSB DFU · apj/hex parsers · manifest · board-ID gates
src/features/         connect · dashboard · setup · calibration · motors · params · firmware · debug · guide
src/store/            connection (owns the per-connection MavSession) · navigation · activityLog
src/i18n/             en / zh / ko / ja
```

Each connection builds a fresh transport → router → ParamStore → Telemetry bundle (`MavSession`); disconnect tears it all down (the router is single-shot per USB-open generation). Feature pages depend only on the core layer's public interfaces.

## Getting started

```bash
npm install
npm run dev        # local dev server
npm test           # unit tests (Vitest)
npm run build      # typecheck + production build
npm run lint
```

### Testing against ArduPilot SITL

Integration tests that need a live flight controller are gated behind `SITL=1`:

```bash
# 1. start SITL from the flight_controller submodule (see docs/notes/sitl.md)
# 2. run the WebSocket↔TCP bridge
node tools/sitl-bridge.mjs
# 3. run the SITL-gated tests
SITL=1 npm test
```

## Firmware distribution

Firmware and a `manifest.json` are **mirrored into this site** at `public/firmware/` and served same-origin — GitHub Releases' asset CDN sends no CORS headers, so the browser can't fetch release binaries directly (verified; see `docs/notes/releases-cors-spike.md`). The authoritative manifest is generated by [`flight_controller`](https://github.com/novaX-ALUX/flight_controller)'s release pipeline (`scripts/gen_manifest.py`), and `scripts/sync-firmware.sh` pulls a tag's assets into `public/firmware/`.

## Deployment

Static site, deployed to **GitHub Pages** at base path `/configurator/` (`novax-alux.github.io/configurator/`). `.github/workflows/deploy.yml` builds and publishes on push to `main`. Web Serial / WebUSB require HTTPS, which Pages provides.

## Before shipping

Three human-owned items remain before a public release:

1. **Real-hardware verification** — the full AF-F4 nano checklist (connect, parameter write/read-back, accelerometer 6-face, compass review gate, **motor test with props off and all six kill-switches**, firmware normal-update + DFU recovery). SITL proves the protocol; it can't prove the destructive paths or that motors actually stop.
2. **LGPL sign-off** — `mavlink-mappings` is LGPL. It's isolated behind `src/core/mavlink/defs.ts` so it can be swapped for a self-generated (MIT-licensed) definitions layer, but shipping it as-is needs a licensing decision and a licenses page. See `docs/notes/decisions-m1.md`.
3. **Deploy verification** — confirm the GitHub Pages build end-to-end and run `scripts/sync-firmware.sh` to populate the firmware mirror.
